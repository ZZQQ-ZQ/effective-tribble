# Skill Gate — 技能创建门禁

> 🔴 创建任何新技能前强制执行。不经门禁直接创建=违规。

## 一、核心原则

**技能太多 → 判断变慢 → 触发不精准。在源头拦截重复，让技能库保持精简。**

## 二、强制执行流程

### 🔴 0号门：触发拦截

用户消息中出现以下表达时，在进入 skill-creator 之前先执行本门禁：

| 触发词 | 优先级 |
|--------|--------|
| "创建技能" / "新建技能" / "写个技能" / "做个技能" | 🔴 强制 |
| "技能化" / "封装成技能" / "加到技能里" | 🔴 强制 |
| "设计一个技能" / "写一个skill" | 🔴 强制 |

### 门1：信息收集

在开始扫描前，确认三要素：

1. **技能方向/名称** — 一句话概括
2. **核心功能** — 它做什么
3. **触发场景** — 用户说什么会用到它

如用户未提供足够信息，追问补充。不问就扫=乱扫。

### 门2：三层扫描

```
工作区 skills/   → 用户全局 skills/   → 系统内置 skills/
（可读写）          （可读写）              （只读，仅参考）
```

扫描命令模板：
```bash
for f in $(find ~/.openclaw/workspace/skills ~/.openclaw/skills -name "SKILL.md" 2>/dev/null); do
  echo "=== $(basename $(dirname $f)) ==="
  head -10 "$f" | grep -E "^name:|^description:"
done
```

### 门3：匹配判定

按四级判定出结果：

| 等级 | 条件 | 操作 |
|------|------|------|
| 🔴 高度重叠 | 功能+触发场景同时匹配已有技能 | **拦截** → 整合进已有技能 |
| 🟡 部分重叠 | 功能匹配但触发场景不同 | 提示选择：扩建旧技能 or 独立创建 |
| 🟢 互补 | 不同但可协作 | 放行，新建时注明协作关系 |
| ⚪ 无关联 | 完全独立 | 放行 |

详细判定规则 → [references/overlap-rules.md](references/overlap-rules.md)

### 门4A：拦截→整合（🔴高度重叠时）

1. 读已有技能 SKILL.md 全文 → 理解现有结构
2. 判断新知识归属：
   - 核心流程 → 写入旧技能 SKILL.md 正文
   - 参考细节 → 追加到旧技能 references/
   - 脚本工具 → 追加到旧技能 scripts/
3. 更新旧技能 frontmatter description → 加入新触发场景
4. 更新技能地图 → [references/skill-map.md](references/skill-map.md)
5. 更新 MEMORY.md 技能索引
6. 告知用户整合结果：整合到哪个技能、追加了什么

### 门4B：放行→创建（🟢⚪时）

1. 确认通过门禁
2. 将新技能信息预登记到 `references/skill-map.md`
3. 进入 skill-creator 标准流程：`init_skill.py → 编辑 SKILL.md → package_skill.py`
4. 创建完成后回写 `references/skill-map.md` 最终确认

## 三、门禁报告格式

每次门禁审查输出以下格式报告：

```
🛡️ Skill Gate 门禁报告

📋 待审查技能：[技能名] — [一句话功能]
🔍 已扫描：工作区 X 个 | 用户全局 X 个 | 系统内置 X 个

🔴/🟡/🟢/⚪ 判定：[等级]

[🔴时]
⚠️ 与已有技能 [技能名] 高度重叠
   重叠维度：[具体说明]
   建议操作：整合进 [已有技能]，追加 [具体内容]
   是否执行整合？

[🟢/⚪时]
✅ 无冲突，放行。
   将与 [相关技能] 协作（如有）。
   进入 skill-creator 创建流程。
```

## 四、安全约束

- 扫描阶段**只读**，不修改任何文件
- 整合操作前**备份**原有技能文件到 `memory/skill-gate-backups/`
- 整合后保留完整操作记录
- 系统内置技能（/usr/lib/）只扫描不修改

## 五、维护

技能地图 `references/skill-map.md` 需在每次新建/整合技能后同步更新。
